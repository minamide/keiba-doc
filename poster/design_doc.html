アプリケーション概要

党員が連携してポスター・チラシの配布活動を効率化するためのWEBアプリケーション。ポスター掲載場所の地図管理、印刷物の在庫管理、党員間の物資やり取り支援を主な機能として提供する。

🗺️ 地図管理
ポスター掲載済み・掲載予定場所を地図上でピン管理。党員全員がリアルタイムで共有。
📦 在庫管理
各党員が保持するポスター・チラシの種類と枚数を管理。全体の在庫状況を可視化。
🔄 物資やり取り
党員間でのポスター・チラシの受け渡しリクエストと承認フローを提供。

認証フロー概要

認証サービスは独立したWorkersとして別ドメインにデプロイ。OAuth2 / OIDCプロバイダとして機能し、メインアプリ以外の将来のアプリからも利用可能。

認証フロー（二次承認あり）
  1. ユーザーがメインアプリにアクセス → 認証サービスにリダイレクト
  2. Google OAuth2 or Email/Password でサインアップ
  3. BetterAuth がユーザーを pending ステータスで登録
  4. ユーザーに「管理者にLINE/Signalで連絡し承認を依頼してください」と案内
  5. 管理者が管理画面からユーザーを approved に変更
  6. ユーザーが再度サインインし、JWTトークンを取得
  7. メインアプリがトークンを検証しアクセスを許可
独立サービスの利点： 将来的に別のWEBアプリを追加する際も、同じ認証サービスのOIDCエンドポイントに接続するだけでシングルサインオンを実現できる。

管理者設定：二次承認の切り替え

管理者は管理画面から二次承認フローの有効/無効を切り替えられる。KVに設定値を保存し、サインアップ時の動作を制御する。

// KVに保存する管理者設定
{
  "require_secondary_approval": true,   // 二次承認の有無
  "approval_contact_method": "line",   // "line" | "signal"
  "admin_contact_info": "@admin_line_id"  // 連絡先（表示用）
}

// 無効時: サインアップ後すぐに approved になる
// 有効時: pending で登録 → 管理者が手動で approved に変更

materials テーブル（印刷物マスタ）

CREATE TABLE materials (
  id                TEXT    PRIMARY KEY,
  name              TEXT    NOT NULL,      -- 例: "A3ポスター2024ver"
  type              TEXT    NOT NULL,      -- "poster" | "flyer"
  description       TEXT,
  thumbnail         TEXT,                  -- R2のURL
  distribute_from   INTEGER,              -- 配布開始日時 (UNIX timestamp, NULL=即時)
  distribute_until  INTEGER,              -- 配布終了日時 (UNIX timestamp, NULL=無期限)
  is_active         INTEGER NOT NULL DEFAULT 1,
                                          -- 1=有効 / 0=無効（手動フラグ）
  created_at        INTEGER NOT NULL,
  updated_at        INTEGER NOT NULL
);

-- 有効判定ビュー（is_active=1 かつ 配布期間内）
CREATE VIEW active_materials AS
  SELECT * FROM materials
  WHERE is_active = 1
    AND (distribute_from  IS NULL OR distribute_from  <= strftime('%s', 'now'))
    AND (distribute_until IS NULL OR distribute_until >= strftime('%s', 'now'));
有効フラグと配布期間の関係：一般党員の在庫登録・やり取り画面には active_materials ビューの結果のみ表示される。 is_active = 0（手動無効化）または配布期間外のいずれかに該当すると非表示になる。既存の在庫・やり取り履歴データは保持される。

inventories テーブル（党員在庫）

在庫の計算モデル：
全体数 = initial_quantity（手動入力）+ 受け取り累計（transfer_requests）
現在庫 = 全体数 − 渡し累計（transfer_requests）− 一般配布済み累計（distribution_logs）
バー進捗 = 一般配布済み ÷ 全体数
党員間やり取りの「渡す」は在庫を減らし「受け取る」は増やす。一般への配布は在庫を減らして配布数にカウントする。 
CREATE TABLE inventories (
  id               TEXT    PRIMARY KEY,
  user_id          TEXT    NOT NULL,
  material_id      TEXT    NOT NULL REFERENCES materials(id),
  initial_quantity INTEGER NOT NULL DEFAULT 0,
                                    -- 手動入力した初期在庫数（やり取り分は含まない）
  updated_at       INTEGER NOT NULL,
  UNIQUE(user_id, material_id)
);

-- 在庫サマリービュー（在庫数・全体数・一般配布済み数を一括集計）
CREATE VIEW inventory_summary AS
  SELECT
    i.user_id,
    i.material_id,
    -- 全体数 = 初期在庫 + 受け取り累計
    i.initial_quantity
      + COALESCE(tr_in.received, 0)                        AS total_quantity,
    -- 現在庫 = 全体数 - 渡し累計 - 一般配布済み
    i.initial_quantity
      + COALESCE(tr_in.received, 0)
      - COALESCE(tr_out.sent, 0)
      - COALESCE(dl.distributed, 0)                        AS current_quantity,
    COALESCE(dl.distributed, 0)                            AS distributed_quantity,
    COALESCE(tr_in.received, 0)                            AS received_quantity,
    COALESCE(tr_out.sent, 0)                               AS sent_quantity
  FROM inventories i
  LEFT JOIN (
    SELECT to_user_id AS user_id, material_id,
           SUM(quantity) AS received
    FROM transfer_requests WHERE status = 'completed'
    GROUP BY to_user_id, material_id
  ) tr_in  ON tr_in.user_id  = i.user_id AND tr_in.material_id  = i.material_id
  LEFT JOIN (
    SELECT from_user_id AS user_id, material_id,
           SUM(quantity) AS sent
    FROM transfer_requests WHERE status = 'completed'
    GROUP BY from_user_id, material_id
  ) tr_out ON tr_out.user_id = i.user_id AND tr_out.material_id = i.material_id
  LEFT JOIN (
    SELECT user_id, material_id, SUM(quantity) AS distributed
    FROM distribution_logs
    GROUP BY user_id, material_id
  ) dl     ON dl.user_id     = i.user_id AND dl.material_id     = i.material_id;

distribution_logs テーブル（一般配布記録）

CREATE TABLE distribution_logs (
  id           TEXT    PRIMARY KEY,
  user_id      TEXT    NOT NULL,              -- 配布した党員
  material_id  TEXT    NOT NULL REFERENCES materials(id),
  quantity     INTEGER NOT NULL,              -- 配布枚数（正の整数）
  distributed_at INTEGER NOT NULL,           -- 配布日時 (UNIX timestamp)
  target_type  TEXT    NOT NULL,             -- "street" | "event" | "door" | "other"
  target_label TEXT,                         -- 配布対象の名称（例: "渋谷駅前街頭"）
  pin_id       TEXT REFERENCES map_pins(id), -- 紐づけた地図ピン（任意）
  lat          REAL,                         -- ピン未紐づけ時の手動緯度
  lng          REAL,                         -- ピン未紐づけ時の手動経度
  note         TEXT,                         -- メモ・備考
  created_at   INTEGER NOT NULL
);
在庫自動操作のトランザクション：
  • POST /distributions 登録時 → inventories.initial_quantity は変えず、distribution_logs に追記。inventory_summary ビューで current_quantity が自動減算される。
  • POST /transfers/:id/approve 承認時 → transfer_requests.status を completed に更新。ビューが受け取り・渡し累計を再集計し在庫が増減する。
  • 在庫不足（current_quantity < 配布枚数）の場合は 422 INSUFFICIENT_STOCK を返す。

transfer_requests テーブル（物資やり取り）

CREATE TABLE transfer_requests (

  message       TEXT,

);

画面遷移フロー

① Email / Password 登録フロー

  1. ログイン画面 → 「新規登録」タップ
  2. サインアップ画面 — 名前・党員番号・都道府県・選挙区・Email・Password を入力
  3. メール確認リンク送信 → ユーザーがメールを確認
  4. BetterAuth が pending ステータスでユーザー登録
  5. 承認待ち画面 — LINE / Signal で管理者に連絡するよう案内
  6. 管理者が管理画面でユーザーを approved に変更
  7. ログイン画面 → サインイン → ダッシュボードへ

② Google認証 登録フロー（初回）

  1. ログイン画面 → 「Googleでサインイン」タップ
  2. Google OAuth 認可画面 → 同意
  3. 初回判定：BetterAuth が member_number 未設定を検出
  4. 初回プロフィール補完画面 — 名前・メールは自動表示、党員番号・都道府県・選挙区のみ入力
  5. BetterAuth が pending ステータスでユーザー登録
  6. 承認待ち画面 → 管理者が承認 → ダッシュボードへ

③ Google認証 ログインフロー（2回目以降）

  1. ログイン画面 → 「Googleでサインイン」タップ
  2. Google OAuth 認可 → BetterAuth がセッション発行
  3. ステータス確認：approved → ダッシュボードへ / pending → 承認待ち画面
